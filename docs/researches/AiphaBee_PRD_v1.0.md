# 0. 执行摘要

## 0.1 一句话产品定义

**AiphaBee 是以 30 多年港股标准化数据为底座、以证据可追溯为核心、同时通过 Web Agent 与 MCP 对外提供研究能力的港股研究操作系统。**

它不应被定义为“港股聊天机器人”，而应被定义为三层产品：

1. **C 端研究产品：** 用户在 Web 上用自然语言完成查数、比较、筛选、事件研究、公告阅读和研究沉淀。
2. **AI 原生数据产品：** 用户可把 AiphaBee Remote MCP 接入其常用 AI 客户端，由外部 Agent 调用标准化港股数据与确定性分析工具。
3. **可商业化的数据治理平台：** 用统一的证券主表、指标定义、授权策略、用量账本和证据链支撑个人订阅、开发者套餐及未来 B2B 授权。

## 0.2 本 PRD 的七项核心决策

1. **一个金融 Tool Registry，两个产品入口。** Web Agent 和 MCP 不各建一套数据接口；二者必须调用相同的领域工具、指标定义、权限规则和证据服务。
2. **MCP 是独立的数据分发产品，不是 Web 的附属接口。** Web 展示权、API/MCP 再分发权、原始数据导出权和派生数据权必须分别确认、分别计费、分别管控。
3. **首发定位为研究与分析工具，而非个性化投资顾问。** MVP 不提供“应该买/卖多少”“目标仓位”“适合你的组合”等个性化建议，不接交易执行。
4. **证据优先于语言表现。** 每个数字、图表和结论都要带 `as_of`、数据延迟、来源记录、数据版本、方法论和必要警告。
5. **原子数据工具与确定性分析优先。** MCP P0 不提供任意 SQL、不允许任意 URL 抓取，也不默认在每个 MCP 工具内部再调用 LLM。
6. **Postgres 作为业务与数据主真值源。** Durable Objects 用于会话协调、幂等、流恢复与热限流；Workflows/Queues 用于深度报告、事件研究、提醒和数据回填。
7. **先解决授权与数据契约，再写大规模产品代码。** “数据合作方有 30 年数据”不自动等于“可以向终端用户或 MCP 客户端再分发”。这是项目 Gate 0。

## 0.3 建议的 MVP 产品面

**Web P0：** 自然语言研究、个股工作台、行情与财务趋势、两到五只股票比较、公告检索与摘要、基础筛选器、保存研究、来源证据、订阅与配额。

**MCP P0：** 证券解析、证券档案、历史行情、延迟/收盘快照、公司行动、财务事实、财务比率、公告检索、股票筛选、证券比较、收益风险计算、数据血缘与权益查询。

**暂不进入 P0：** 实时逐笔/盘口、衍生品、自动交易、组合调仓建议、任意批量全市场下载、任意 SQL、社区荐股、开放式用户代码执行。

## 0.4 上线前四个硬性门槛

| 门槛 | 必须回答的问题 | 不满足时的处理 |
|---|---|---|
| 数据授权 | 哪些字段可用于 Web、MCP、API、导出、缓存和派生？ | 对未确认字段默认拒绝分发 |
| 监管分类 | 具体功能是否构成证券投资建议或 Type 4 受规管活动？ | 取消个性化建议或接入持牌合作方 |
| 数据质量 | 证券主表、复权、财务重述、退市样本、交易日历是否经过黄金样本验证？ | 不开放相应工具或标注 Beta |
| 可审计性 | 能否重放一次答案使用的工具输入、数据版本、模型版本和输出？ | 不允许进入公开付费阶段 |

# 1. 文档背景、依据与边界

## 1.1 已确认背景

- 产品名称：AiphaBee。
- 产品方向：面向 C 端、高并发的港股分析 Agent，同时提供 MCP 数据与分析能力。

# 2. 产品愿景与战略定位

## 2.1 产品愿景

让个人投资者和其所使用的 AI Agent，都能像调用专业金融终端一样，获得**长期、结构化、解释清楚、可验证、可复用**的港股研究能力。

## 2.2 市场切入点

AiphaBee 不与大型券商在交易入口上正面竞争，也不与通用大模型在“什么都能聊”上竞争。它的切入点是：

> **港股本地深度 × 30 年历史连续性 × 每一结论可追溯 × Web/MCP 双渠道。**

当前金融 AI 产品大致分为四类：券商内置 AI、财经门户 AI、全球金融工作台和通用大模型。券商 AI 通常免费或与交易账户绑定，优势是交易闭环；全球工作台强调多数据源与机构工作流；OpenBB 等产品已把 MCP 视为金融数据进入 AI 工作流的重要接口。[^openbb] AiphaBee 的差异化不应是“也有一个聊天框”，而应是港股数据深度、方法论一致性和可审计输出。

## 2.3 品牌承诺

1. **任何数字都能追溯。**
2. **任何时间序列都说明复权和时间口径。**
3. **事实、计算、推断和观点明确分层。**
4. **不知道时明确说不知道，不用语言流畅掩盖数据缺失。**
5. **用户在 Web、MCP 或未来 API 中得到相同口径的结果。**

## 2.4 产品原则

| 原则 | 产品含义 |
|---|---|
| Evidence before eloquence | 先取得并验证数据，再生成自然语言；不允许模型凭记忆补金融数字 |
| One truth layer, many channels | Web、MCP、B2B API 共用证券主表、指标定义与权限服务 |
| Data rights by design | 权利元数据进入数据模型和运行时，不靠上线前人工记忆 |
| Deterministic where possible | 收益、风险、同比、筛选、事件研究由代码计算，不交给 LLM 心算 |
| Read-only by default | P0 工具全部只读；任何写入、提醒和外部动作均显式授权 |
| Progressive disclosure | 新手先看到结论和解释，专业用户可展开原始数据、公式和来源 |
| Cost is a product constraint | 每个 run 有工具步数、Token、数据行数、时间范围和成本预算 |
| Neutrality and safety | 不因券商导流、广告或合作方利益改变研究排序和结论 |

“当一只港股今天或最近大幅波动时，我希望快速看到价格、成交、公告、财务和同业背景，从而区分已知事实与市场猜测。”

“当我研究一家公司时，我希望用统一口径查看十年到三十年的股价、收入、利润、现金流、估值与公司行动，以识别周期和结构变化。”

### JTBD-D：比较而非孤立判断

“当我评估一家公司时，我希望与同业、指数或自身历史区间比较，避免只看单一绝对数值。”

### JTBD-E：把数据带到自己的 AI 工作流

“当我使用外部 AI 客户端时，我希望授权它调用可靠的港股工具，而不是复制粘贴网页或让模型猜测数据。”

### JTBD-F：保存和重现研究

“当我形成一个研究结论时，我希望保存当时使用的数据版本、图表和来源，以便以后重放、更新或分享。”

# 4. 目标、非目标与成功定义

## 4.1 产品目标

### 用户目标

- 把完成一次基础港股研究的时间从数十分钟或数小时降至数分钟。
- 让非专业用户理解关键财务概念，同时让专业用户能查看原始数据和口径。
- 让用户在任何支持 Remote MCP 的 AI 客户端中安全调用 AiphaBee 数据。
- 让每次研究可保存、可核查、可更新。

### 业务目标

- 建立可持续的个人订阅收入，而不是只依赖广告或导流。
- 形成 MCP 用量收入和开发者生态，为未来 B2B 数据产品建立入口。
- 把数据合作方的长期数据资产产品化，并通过授权、质量和使用反馈提高资产价值。
- 形成“数据—工具—使用—评估—改进”的复利壁垒。

### 平台目标

- 同一工具在 Web Agent、MCP 和内部评估中返回一致结果。
- 数据权限能够精确到数据集、字段、渠道、用户层级、延迟、时间范围和导出方式。
- 任何用户可见数字都能映射到来源记录和计算方法。
- 模型供应商可替换，业务逻辑不绑定单一 LLM。

## 4.2 非目标

MVP 明确不做：

- 证券交易下单、自动交易、代客执行或券商账户托管。
- 基于用户风险承受能力、资产或目标给出个性化买卖和仓位建议。
- 覆盖所有全球市场、期权、期货、窝轮、牛熊证和高频逐笔策略。
- 为用户执行任意代码、Shell、浏览器自动化或未知第三方插件。
- 提供不受限制的全市场原始数据导出、任意 SQL 或数据库镜像。
- 以社区荐股、排行榜、跟单或高刺激预测作为主要增长机制。
- 宣称 AI 能准确预测股价或保证收益。

## 4.3 北极星指标

**Weekly Verified Research Outcomes（周度已验证研究成果，WVRO）**：一周内满足以下条件的独立研究任务数：

1. 至少成功调用一个金融工具；
2. 最终答案包含可打开的证据或数据来源；
3. 用户完成至少一个高意图动作：保存、加入观察列表、比较、导出允许内容、创建提醒、继续追问或在 MCP 客户端完成后续调用；
4. 任务未触发数据错误、严重幻觉或合规拦截。

该指标比“聊天消息数”更接近真实价值，也能避免通过低质量对话虚增活跃度。

# 5. 产品范围与信息架构

## 5.1 产品模块

### A. Web 研究产品

1. **Ask / 研究对话**：自然语言提问、澄清、工具执行、流式答案、证据卡片。
2. **个股工作台**：公司档案、行情、财务、估值、公告、公司行动、历史事件。
3. **比较器**：多证券、多指标、多时期对比。
4. **筛选器**：自然语言转结构化条件，同时允许用户检查和修改条件。
5. **公告与文件研究**：搜索、定位原文、摘要、关键数字抽取、跨期对比。
6. **事件研究**：围绕业绩、配股、回购、分红等事件计算窗口收益与同业/指数对照。
7. **研究库**：保存问答、图表、证据快照、笔记与版本。
8. **观察列表与提醒**：价格、公告、财务披露和自定义条件提醒。
9. **账户与订阅**：套餐、配额、用量、发票、MCP 连接管理。

### B. MCP 数据与分析平台

1. Remote MCP endpoint：`/mcp`。
2. OAuth 授权和服务端 API Key。
3. 只读工具目录与结构化输出。
4. Developer Console：连接指南、密钥、scope、配额、日志、示例。
5. 版本、弃用、状态页和用量结算。
6. 数据授权与输出限制的运行时执行。

MCP 官方的 Streamable HTTP 传输使用单一 HTTP endpoint，支持 POST/GET，并可选 SSE；实现时还应验证 `Origin` 并为连接启用适当认证。[^mcp-transport] MCP 工具应声明输入/输出 Schema，使客户端能够严格验证和正确解析结构化结果。[^mcp-tools]

### C. 内部平台

1. 数据集与字段授权登记。
2. 证券主表与标识映射管理。
3. 数据质量、对账、回填和更正。
4. Agent run、tool call、证据与模型日志查看。
5. 订阅、用量、成本、退款和异常用户管理。
6. Prompt、工具版本、模型路由、实验与回滚。
7. 合规规则、敏感功能开关、事故处置和审计导出。

## 5.3 首页与首次体验

首次进入不应只显示空聊天框，应展示：

- 全局证券搜索；
- 3–5 个可直接运行的研究模板；
- 当日市场概览和数据延迟说明；
- “为什么今天上涨/下跌”“比较两家公司”“筛选一类公司”“读最新公告”等任务入口；
- 明确的示例问题，分别面向新手和专业用户；
- MCP 用户可直接进入“连接到我的 AI”流程。

# 6. 核心用户旅程

## 6.1 旅程一：解释个股近期波动

1. 用户输入：“腾讯这三天为什么跌？有基本面变化吗？”
2. 系统解析证券与时间范围，显示行情延迟和市场状态。
3. Agent 调用价格、成交、公告、财务日历、同业/指数对照等只读工具。
4. 如果存在证券或时间歧义，先给出可点击澄清，不擅自选择。
5. 系统返回：已知事实、相关事件、对照数据、尚无法证实的市场解释。
6. 用户可展开每项证据、切换复权口径、继续运行“与恒生科技指数对比”。
7. 保存后生成带数据版本的研究快照。

**成功条件：** 用户能够在答案中区分“公司披露”“客观市场数据”“模型推断”和“未知”。

## 6.2 旅程二：长期基本面与股价对照

1. 用户打开个股页或输入：“过去十五年港交所收入、利润率和股价回撤有什么关系？”
2. 系统确认指标定义、财务年度、币种、复权方式和对照基准。
3. 确定性分析引擎计算时间序列、回撤和相关统计。
4. Agent 解释不同阶段，但不把相关性表述成因果。
5. 用户可查看每一年度原始财务事实、重述版本和公告来源。

## 6.3 旅程三：自然语言筛选

1. 用户输入：“找出过去五年收入持续增长、净现金、股息率超过 3%、市值 100 亿港元以上的港股。”
2. 系统把语言转成可编辑的结构化条件。
3. 页面在执行前显示字段定义、日期与缺失值处理方式。
4. 工具返回结果、命中理由、未命中原因和分页游标。
5. 用户保存筛选或创建定期提醒。

## 6.4 旅程四：通过 MCP 使用数据

1. 用户在 AiphaBee 的“数据接入”页面创建连接。
2. 选择目标客户端和授权范围，例如 `market.read`、`fundamentals.read`、`filings.read`。
3. 外部客户端完成 OAuth 授权，并发现可用工具。
4. 用户在外部 AI 中问：“比较 0700、9988 和 3690 最近十年的收入增速和最大回撤。”
5. 外部 AI 调用 AiphaBee 工具；AiphaBee 返回结构化数据、来源、配额和警告。
6. 外部 AI 自行生成回答。AiphaBee 不默认再调用一次内部 LLM，避免成本、延迟和不可预测性叠加。
7. 用户可在 Developer Console 查看该次工具调用、消耗和响应摘要。

## 6.5 旅程五：深度报告

1. 用户选择“深度研究”并设定公司、范围和模板。
2. 系统预估 Bee Credits、数据范围和预计输出，用户确认。
3. 请求进入 Workflow，按步骤获取数据、执行确定性分析、生成章节、做引用校验。
4. 用户离开页面后任务继续，完成后发送站内/邮件通知。
5. 报告带完整证据索引、版本和重新运行按钮。

# 8. Agent 行为与答案规范

## 8.1 Agent 运行上下文

每次 run 必须包含：

- `run_id`、`user_id`、`workspace_id`；
- 套餐、数据权益、地区、语言和时区；
- 允许的工具集合与工具版本；
- 最大步骤、最大 Token、最大数据行数、最大工具成本；
- 模型层级和 fallback 策略；
- 是否允许长任务、导出和保存；
- 数据授权策略版本；
- 风险模式：普通研究、敏感建议、可能误导、隐私数据等。

## 8.2 规划和停止规则

**交互式 run 默认建议：**

- 最多 6–8 个工具步骤；
- 一次调用最多并行 3 个相互独立的只读工具；
- 超出交互预算的事件研究、长文档或多公司深度报告转 Workflow；
- 发生连续两次同类工具错误时停止自动重试；
- 没有足够证据时返回缺口，而不是继续扩大搜索；
- 到达费用预算时返回已完成部分、未完成部分及继续成本。

AI SDK v7 的 `ToolLoopAgent` 适合可复用、多步、流式工具循环；耐久和可恢复的长流程可交由 `WorkflowAgent` 或 Cloudflare Workflows。[^ai-sdk-agent] Cloudflare Workers 的 HTTP 请求在客户端保持连接时没有固定墙钟上限，但 CPU、内存、并发连接和 subrequest 仍有限制，因此“能保持流式连接”不等于适合把所有深度任务放在单次请求中。[^workers-limits]

## 8.3 答案结构

默认答案按以下顺序生成：

1. **直接回答：** 用 2–5 句回答用户问题，不先堆砌过程。
2. **时间与数据状态：** `as_of`、市场时区、实时/延迟/收盘、数据覆盖。
3. **关键证据：** 3–6 项最相关数据、图表或公告。
4. **解释：** 明确哪些是事实、确定性计算和推断。
5. **反证与风险：** 指出可能的替代解释、样本限制和缺失数据。
6. **来源与方法：** 可展开来源记录、公式、复权、币种和版本。
7. **下一步：** 最多给 2–3 个真正相关的后续研究动作。
8. **适当免责声明：** 不用免责声明代替功能边界和合规控制。

## 8.4 证据强度，而非伪精确“信心分数”

不建议默认显示“87% 可信”之类未经校准的数字。建议显示：

- **证据强：** 直接来自公司披露、交易数据或一致的确定性计算；
- **证据中：** 多项数据支持，但因时间、口径或因果关系存在限制；
- **证据弱：** 主要是市场叙事、单一来源或间接推断；
- **无法判断：** 数据缺失、冲突或授权不可用。

## 8.5 禁止行为

- 在没有工具证据时生成具体价格、财务数字、日期或公告内容。
- 将相关性写成因果关系。
- 把历史表现描述为未来保证。
- 绕过用户套餐、地区或数据授权调用工具。
- 把公告、网页或用户输入中的指令当作系统指令。
- 自动给出个性化仓位、买卖时点或确定性涨跌预测。
- 隐藏数据延迟、重述、币种和复权口径。

# 9. MCP 工具产品设计

## 9.1 工具设计原则

1. **稳定、窄而清晰。** 一个工具完成一个明确任务。
2. **不暴露数据库结构。** 用户调用领域概念，不依赖内部表名。
3. **确定性输出。** 数据查询和计算不依赖 LLM 自由生成。
4. **支持小上下文。** 默认只返回完成任务所需字段，避免把巨量 JSON 塞给模型。
5. **可分页、可预算。** 时间范围、证券数量、字段和行数都有上限。
6. **证据原生。** 每个结果都有数据状态和 lineage。
7. **权限原生。** 工具描述可见不代表所有数据字段都可用；响应按用户权益裁剪。
8. **向后兼容。** 不随意改名、改单位或改变默认口径。

- `run_event_study`：事件窗口、基准、异常收益和样本统计。
- `get_market_breadth`：上涨/下跌、成交集中度、行业宽度等。
- `get_ownership_and_short_selling`：在授权允许时提供持股和沽空数据。
- `get_buybacks_and_placements`：回购、配股、供股等资本事件的结构化分析。
- `get_consensus_or_estimates`：只有在明确拥有再分发权时开放。
- `get_portfolio_analytics`：仅处理用户主动提供并授权的持仓；不输出个性化交易建议。
- `create_alert`：写操作，必须有显式用户确认、独立 scope 与幂等键。

## 9.4 不建议提供的工具

- `run_sql`、`query_anything`、`download_all_market_data`。
- 任意 URL 抓取或任意浏览器操作。
- 直接返回完整内部数据库表。
- 没有范围上限的批量工具。
- 名为 `predict_stock_price`、`get_buy_signal` 的伪确定性工具。
- 把“检索数据 + 调 LLM + 生成观点”全部封装为不可解释的黑箱工具。

## 9.6 标准错误码

| 错误码 | 含义 | 客户端建议 |
|---|---|---|
| `AUTH_REQUIRED` | 未认证或 Token 失效 | 重新授权 |
| `SCOPE_DENIED` | 缺少工具 scope | 请求用户批准额外 scope |
| `DATA_NOT_LICENSED` | 套餐或合同不允许该数据 | 缩小字段/范围或升级套餐 |
| `SYMBOL_AMBIGUOUS` | 证券解析存在多个候选 | 展示候选并让用户选择 |
| `OUT_OF_RANGE` | 时间或证券数量超限 | 缩小请求 |
| `TOO_MANY_ROWS` | 响应超过行数限制 | 使用游标或聚合粒度 |
| `RATE_LIMITED` | 速率或并发超限 | 按 `retry_after` 重试 |
| `BUDGET_EXCEEDED` | 当期用量或单任务预算不足 | 用户确认加购或缩小任务 |
| `UPSTREAM_STALE` | 上游数据超过 SLA | 显示最后可用时点，不静默返回旧数据 |
| `DATA_QUALITY_HOLD` | 数据被质量规则隔离 | 稍后重试或使用替代口径 |
| `INTERNAL_ERROR` | 未分类系统错误 | 携 request_id 联系支持 |

## 9.7 OAuth scope 建议

- `security.read`
- `market.read`
- `fundamentals.read`
- `filings.read`
- `analytics.run`
- `portfolio.read`（P1，需额外隐私说明）
- `alerts.write`（P1，写操作）
- `exports.read`（高风险，单独授权）
- `admin.usage.read`（团队管理员）

不得把第三方客户端 Token 直接透传给数据合作方。AiphaBee 是资源服务器和授权执行点，所有调用都映射到自己的用户、Workspace、scope、套餐与审计记录。

# 10. 数据产品与模型

## 10.1 数据产品化流水线

![数据产品化流水线](/mnt/data/aiphabee_prd_assets/data_pipeline.png){width=75%}

底层合作方数据不应直接暴露给 Agent。建议经过：原始快照、标准化、质量对账、派生指标、Serving Store、Data Access Gateway，再进入 Web/MCP。

## 10.2 核心数据域

| 数据域 | 核心实体/事实 | 关键口径 |
|---|---|---|
| 证券主表 | company、instrument、listing、identifier_history | 公司与证券分离；代码有有效期 |
| 交易日历 | market_calendar、session | 香港时区、半日市、恶劣天气安排 |
| 行情 | quote_snapshot、price_bar | 原始/复权、币种、延迟、交易状态 |
| 公司行动 | dividend、split、rights、placement、buyback | 公告日、除权日、生效日、支付日分离 |
| 财务事实 | statement、financial_fact、segment_fact | 期间、披露日、币种、单位、重述版本 |
| 派生指标 | metric_value、metric_definition | 公式与版本、TTM、分母异常处理 |
| 公告文件 | filing、document、document_chunk | 语言、发布日期、页码/段落、来源哈希 |
| 市场统计 | breadth、short_selling、ownership | 数据来源、频率、授权等级 |
| 研究证据 | evidence_snapshot、citation | 输入、输出、来源记录、方法版本 |
| 商业与权限 | plan、entitlement、rights_policy、usage_ledger | 渠道、字段、时间范围、导出、费用 |

## 10.3 时间与版本模型

金融数据至少需要以下时间字段：

- `period_start` / `period_end`：财务事实所属期间；
- `published_at`：市场首次可知时间；
- `effective_at`：公司行动或规则生效时间；
- `ingested_at`：AiphaBee 收到数据的时间；
- `restated_at`：重述或更正时间；
- `valid_from` / `valid_to`：证券名称、行业、标识等历史有效期；
- `data_version`：可复现的数据批次版本。

用户问“截至某日当时知道什么”时，系统必须按 `published_at` 做 point-in-time 查询，避免未来数据泄漏。

## 10.4 复权与总回报

价格工具至少提供：

- `raw`：原始交易价格；
- `split_adjusted`：拆股、合股等数量性公司行动复权；
- `total_return_adjusted`：计入现金分红与再投资假设的总回报序列。

每种口径必须说明：

- 处理哪些公司行动；
- 使用前复权还是后复权；
- 分红再投资时间与价格假设；
- 停牌、退市和缺失交易日处理；
- 方法论版本。

## 10.5 财务事实与重述

不得用“最新值覆盖旧值”。同一公司、同一期间、同一指标可能存在：首次披露、后续重述、会计准则调整和数据供应商修订。模型应保留：

- reported value；
- standardized value；
- reported currency、unit、scale；
- accounting standard；
- filing/document reference；
- version and restatement reason；
- 是否由 AiphaBee 派生。

## 10.6 指标定义库

每个指标必须包含：

- 唯一 `metric_id` 和中英文名称；
- 公式和依赖事实；
- 适用行业；
- 时间口径（年度、季度、TTM、点时）；
- 币种和缩放；
- 缺失值与负分母处理；
- 方法版本和生效时间；
- 用户可读解释与示例。

## 10.7 数据质量框架

### 自动规则

- 主键、时间、币种和单位完整性；
- OHLC 关系、成交量、价格跳变和重复记录；
- 公司行动前后价格/股本一致性；
- 三表勾稽与财务恒等式；
- 财务同比异常、数量级变化和币种切换；
- 公告发布日期与报告期逻辑；
- 跨源或合作方对账；
- 退市、长期停牌、代码变更和双重上市特殊规则。

### 黄金样本

上线前建立至少以下黄金样本：

- 50–100 只跨行业、跨年代、含退市/复牌/供股/拆股的证券；
- 20 个复杂公司行动案例；
- 20 个财务重述案例；
- 10 个代码或名称历史变更案例；
- 10 个多币种或双重上市案例；
- 主要指数与行业基准的历史成分样本。

## 10.8 数据更正

当发现错误时：

1. 隔离受影响数据；
2. 记录原值、新值、原因、来源与影响范围；
3. 重新计算派生指标；
4. 标记受影响的证据快照和已保存报告；
5. 对重大错误通知用户；
6. 保留审计记录，不直接抹掉历史。

# 11. 目标技术架构

## 11.1 总体架构

![AiphaBee 目标架构](/mnt/data/aiphabee_prd_assets/architecture.png){width=96%}

现有研究简报已决定将 Agent 层置于 Hono + AI SDK v7 on Cloudflare Workers，并用 Workflows/DO/Queues/Cron 承载长任务和状态协调。[^brief] 本 PRD 在此基础上增加三个关键产品化组件：

1. **共享 Tool Registry：** Web Agent 和 MCP 的唯一业务能力入口。
2. **Data Access Gateway：** 统一执行数据授权、响应裁剪、行数/时间限制和证据注入。
3. **Evidence & Lineage Service：** 把一次工具调用与来源记录、数据版本、方法论和用户可见引用连接起来。

## 11.2 双入口请求流

![双入口请求流](/mnt/data/aiphabee_prd_assets/flows.png){width=88%}

### Web Agent 路径

Web 请求进入 Agent Runtime，由模型在有限步骤内选择工具。工具结果先形成结构化证据，再由模型生成解释。LLM 是“解释与编排层”，不是金融事实数据库或计算器。

### MCP 路径

外部客户端经过 OAuth/API Key 与 Data Access Gateway 后直接调用同一 Tool Registry。原子工具默认不触发内部 LLM。这样可获得：

- 更低延迟和成本；
- 稳定、可缓存、可测试的输出；
- 避免“外部 LLM 调内部 LLM”的嵌套不可控；
- 更清晰的数据授权和计费。

## 11.3 组件职责

| 组件 | 建议技术 | 职责 |
|---|---|---|
| Web UI | TanStack Start + Vite | SSR、SPA、对话、图表、工作台、账户 |
| Edge Gateway | Hono on Workers | Web API、MCP、鉴权、策略、限流、配额、审计 |
| Agent Runtime | AI SDK v7 | 多步工具循环、流式、模型路由、结构化输出 |
| Long-running Orchestrator | Workflows | 深度报告、事件研究、长文档、多步可恢复任务 |
| Per-run Coordination | Durable Objects | 幂等、SSE/会话协调、并发与热状态 |
| Shared Tool Registry | TypeScript domain services | 工具 Schema、版本、权限、执行与测试 |
| Data Access Gateway | Service binding/domain layer | 权利执行、字段裁剪、范围限制、缓存与 lineage |
| System of Record | Postgres/Supabase via Hyperdrive | 用户、权益、账单、审计、证券与财务事实 |
| Object Store | R2 | 原始文档、不可变报告快照、允许的导出 |
| Search | pgvector first；Vectorize optional | 公告/文件检索与语义召回 |
| Config/Cache | KV / Cache API | 功能开关、公共元数据和短时缓存 |
| Model Control | Cloudflare AI Gateway | 日志、成本、限流、缓存、重试、fallback |
| Observability | OTel + logs + eval store | 性能、成本、质量、安全、数据 SLA |

Cloudflare Durable Objects 适合需要单实体协调、强一致和持久连接的场景，不应被用作所有无状态请求的默认处理器。[^do] Hyperdrive 可连接现有 Postgres，并在数据库附近维护连接池，适合 Workers 的短生命周期连接模式。[^hyperdrive] AI Gateway 可提供模型调用分析、日志、缓存、限流、重试和 fallback。[^ai-gateway]

## 11.4 存储选型建议

### 推荐默认组合

- **Postgres/Supabase：主真值源。** 原因是数据关系复杂、需要事务、RLS/权限、用量账本、审计、财务 point-in-time 查询和丰富 SQL。
- **Hyperdrive：Workers 到 Postgres 的连接与缓存层。**
- **Durable Objects：按 run、用户或连接进行协调。** 不作为全局财务数据库。
- **R2：公告原文、原始批次、报告快照和大对象。**
- **KV：功能开关、少量公共配置和容忍最终一致的缓存。**
- **pgvector：首发文件检索。** 权限、关系和元数据过滤可与 Postgres 一体处理。
- **Vectorize：可作为公共公告大规模边缘检索的后续优化。**

### 不建议

- 用 D1 作为首发全部业务与金融事实的唯一主库；
- 把会话历史、权益和计费分散到多个没有统一事务的 KV；
- 在每次用户工具调用时实时直连合作方生产库；
- 把所有原始文档和长 JSON 存入 Worker 内存或聊天消息。

## 11.5 数据合作方接入

推荐采用“产品化数据集”而不是“Agent 直连合作方任意 API”：

1. 双方定义数据契约、字段字典、时间口径、授权与 SLA；
2. 通过批量、增量 CDC 或受控 API 同步到 AiphaBee Raw/Bronze；
3. AiphaBee 负责标准化、版本化、派生指标、质量隔离和 Serving；
4. 每条记录保留合作方 source ID；
5. 合作方数据故障时，AiphaBee 可明确返回最后可用时间，并按 SLA 回传问题；
6. 双方按订阅用户、MCP 用量或数据包进行结算。

## 11.6 模型路由

按任务分层，而不是所有请求都用最贵模型：

- **轻量模型：** 意图识别、证券解析辅助、简单格式化、摘要草稿；
- **主模型：** 复杂研究规划、证据综合、跨文档解释；
- **确定性代码：** 所有金融计算、筛选和结构化转换；
- **fallback：** 主模型超时或限流时切换备选，但必须记录模型变化；
- **缓存：** 仅缓存可安全复用且无用户敏感数据的模型/工具结果。

# 12. 非功能需求

## 12.1 性能与可用性

以下是首发建议目标，需在仓库和真实数据链路核验后调整：

| 指标 | 目标 |
|---|---|
| Web/MCP 核心 API 可用性 | 月度 99.9% |
| 原子 MCP 工具 P95 | 热路径 < 800ms；冷/复杂查询 < 2.5s |
| Web 首 Token P95 | < 2.5s（不含需澄清的交互） |
| 简单研究完成 P95 | < 15s |
| 长任务 | 立即返回 task_id，支持断线恢复和通知 |
| MCP 工具成功率 | > 99.5%，不含用户输入和授权错误 |
| 用量记账延迟 | < 5 分钟；高成本工具同步预扣 |
| 证据链接可用性 | 与报告保留期一致 |

## 12.2 扩展性

- 无状态 Worker 根据请求自动扩展；
- 工具层不得依赖单进程内存会话；
- 数据库连接由 Hyperdrive 池化；
- 高成本工具进入独立并发池和队列；
- MCP 批量请求按 Workspace、用户、工具和数据集多维限流；
- 缓存 key 必须包含数据版本、权限版本、字段集合和口径；
- 对全市场筛选、长区间序列和事件研究使用预计算或分区查询。

## 12.3 可审计性

每次 run/tool call 至少记录：

- 用户、Workspace、Token/client、IP 风险摘要；
- 工具名、版本、输入摘要、授权策略版本；
- 数据集、数据版本、来源记录、缓存命中；
- 模型、提示词版本、Token、成本和延迟；
- 输出哈希、错误、重试与人工干预；
- 用户反馈、后续更正和影响报告。

敏感内容按隐私和保留策略脱敏；审计日志与产品分析日志分离。

## 12.4 可访问性与本地化

- Web 目标 WCAG 2.1 AA；
- 图表有文本摘要、键盘操作和高对比模式；
- 默认香港时区，明确显示币种；
- 代码格式统一为 `00700.HK`，同时兼容本地习惯输入；
- 繁体中文优先完成产品质量，简体中文和英文按同一术语表输出；
- 财务术语提供中英文和口径解释。

# 13. 安全、隐私与滥用防护

## 13.1 主要威胁

- 公告、网页或用户文本中的 prompt injection；
- MCP Token 泄露、scope 过大、连接劫持和 DNS rebinding；
- 通过分页、多个账户或并发绕过数据下载限制；
- 任意 SQL、任意 URL 或超大查询导致数据泄露与成本攻击；
- 把用户持仓、身份或研究内容泄露给模型供应商；
- 数据供应链投毒、错误版本和来源伪造；
- LLM 生成虚假数字或错误引用；
- 共享链接扩大数据授权；
- 付费模型或高成本工具被恶意消耗。

## 13.2 控制措施

### MCP 与 API

- Streamable HTTP endpoint 校验 `Origin`；
- OAuth PKCE、短期 Token、明确 audience 和 scope；
- API Key 哈希存储、可轮换、可限 IP、只显示一次；
- 请求体、响应行数、时间范围、证券数量和并发限制；
- 所有输入/输出 Schema 严格验证；
- 对高风险导出实施单独 scope、二次确认和审计；
- 不允许 Token 透传到数据合作方；
- 按用户、Workspace、客户端、工具、数据集和 IP 风险多维限流。

### Agent 与内容

- 把公告、网页、检索片段标记为“不可信数据”，与系统指令隔离；
- 工具 allowlist，禁止模型构造任意数据库查询；
- 文档解析去除脚本、隐藏文本和可疑指令；
- 关键数字在生成后进行 evidence binding 校验；
- 高风险建议触发政策分类器和安全模板；
- 对外部动作使用 tool approval，P0 不提供交易动作。

### 隐私

- 最小化发送给模型的个人数据；
- 用户持仓和研究历史默认不用于公共模型训练；
- 提供记忆开关、删除、导出和保留期限；
- 模型供应商、日志和支持工具纳入第三方风险评估；
- 生产数据与测试/eval 数据隔离并脱敏。

香港 PCPD 的 AI 个人资料保障框架要求使用涉及个人资料的 AI 系统时采用风险为本、privacy-by-design、治理、实施和沟通措施，并遵守 PDPO 的数据保障原则。[^pcpd]

# 14. 数据授权与监管边界

## 14.1 Gate 0：数据再分发权

HKEX 公开说明区分内部使用与向客户再分发；向客户再分发通常需要 Market Data Vendor Licence，而非仅有 End-user Licence。[^hkex-faq] HKEX 也对非展示使用及不同类别的数据用途设置收费。[^hkex-fee]

因此，AiphaBee 与数据合作方必须共同完成字段级权利矩阵：

| 权利维度 | 必须确认的内容 |
|---|---|
| 数据所有者/来源 | 合作方自有、HKEX、上市公司公告、第三方估算或其他 |
| Web 展示 | 是否可在公开/登录/付费页面展示，是否有用户计数要求 |
| MCP/API 再分发 | 是否允许机器可读返回给终端用户或其 AI 客户端 |
| 原始与派生 | 原始字段、简单计算、深度派生数据的权利是否不同 |
| 实时/延迟/EOD | 最小延迟、标识方式、非展示使用费用 |
| 历史范围 | 30 年历史是否全部可对外，是否限制批量和存储 |
| 导出与缓存 | 客户端是否可保存、缓存、下载、再传播 |
| 用户类型与地区 | 个人、专业投资者、企业、香港/海外是否不同 |
| 设备与订阅者 | 是否需报送用户、设备、应用或月度 subscriber report |
| 审计与终止 | 合作方/HKEX 审计权，合同终止后的删除和停止服务 |
| 商业条款 | 固定费、最低保证、每用户、每调用、收入分成或超额费 |

**默认拒绝原则：** 未被明确允许的字段、渠道和用途不得上线。MCP 返回机器可读数据，尤其容易被存储和再分发，不能把 Web 展示授权自动扩展到 MCP。

## 14.2 证券投资建议与 Type 4 风险

SFC 将 Type 4 列为“就证券提供意见”；其牌照手册列出面向公众出版物/广播的有限豁免，但互动式、个性化 Agent 是否适用不能靠产品自行推断。[^sfc-license] SFC 对持牌机构使用生成式 AI 提供投资推荐、投资意见或投资研究也提出管理层责任、模型风险、网络及数据风险、第三方风险等要求。[^sfc-ai]

### MVP 产品边界建议

- 用“研究”“分析”“数据解释”，不承诺“荐股”或“智能投顾”；
- 可回答历史事实、财务分析、情景和风险，但不输出个性化买卖指令；
- 不收集风险承受能力并据此自动给出适合性结论；
- 不输出目标仓位、自动再平衡、止盈止损指令；
- 不接下单；
- 对“我该不该买”类问题，转为展示可验证的正反证据与需要用户自行判断的因素；
- 上线前取得香港律师/合规顾问对具体页面、提示词、营销文案和收费模式的书面意见。

### 后续两条路线

1. **研究工具路线：** 持续保持一般研究和数据工具边界。
2. **持牌/合作路线：** 与持有相关牌照的机构合作或自建持牌能力，加入适合性、监督、记录和人工介入机制。

免责声明不能替代牌照判断、产品限制和运行时政策执行。

# 15. 商业模式

## 15.1 收入结构

建议形成四条收入线：

1. **B2C Web 订阅：** 个人研究、长期数据、事件研究、提醒和报告。
2. **MCP/Developer 用量：** 结构化数据与分析工具的订阅和超额用量。
3. **Team/Enterprise：** 团队席位、审计、SSO、SLA、私有连接和定制授权。
4. **B2B2C/白标：** 为券商、媒体、理财平台或数据公司提供嵌入式研究工具和 MCP/API。

券商导流、开户分成或广告可作为补充，但必须与研究排序和结论隔离，并清晰披露利益冲突。

## 15.2 建议套餐与价格假设

以下为产品验证价格，不是最终报价；需在数据授权成本和目标市场访谈后调整。

| 套餐 | 建议月价 | 适用用户 | Web 权益 | MCP 权益 |
|---|---:|---|---|---|
| Free | HK$0 | 试用与新手 | 基础问答、有限历史、少量保存 | 极少量轻工具调用；无批量/导出 |
| Plus | HK$88 | 一般个人投资者 | 更多问答、较长历史、观察列表 | 1 个连接、基础数据额度 |
| Pro | HK$228 | 资深个人投资者 | 全 30 年授权历史、比较、筛选、事件研究、深度报告 | 较高额度、全部 P0 工具 |
| Developer | HK$688 起 | AI Power User/个人开发者 | Pro Web 权益 | 多连接/API Key、批量分页、超额计费 |
| Team | HK$3,888 起 | 小型研究团队 | 共享研究、管理和席位 | Workspace、审计、较高并发与额度 |
| Enterprise | 定制 | 券商/媒体/企业 | SSO、SLA、定制功能 | 私有数据、定制工具、商业授权 |

### Free 的原则

Free 的目标是完成价值验证，不是提供可被无限抓取的数据镜像：

- 数据时间范围和调用次数有限；
- 高价值 30 年历史、事件研究和批量工具放在付费层；
- 免费 MCP 只能小额度、低并发、不可用于商业再分发；
- 对异常自动化使用触发验证或暂停。

## 15.3 用量计费

建议采用**加权数据/计算 credits**，而不是简单按一次调用计费。原因是元数据查询、30 年行情、全市场筛选和深度事件研究的成本差异巨大。

### 示例权重

- 证券解析/档案：1 credit；
- 延迟行情快照：1–2 credits；
- 1 年历史行情：2 credits；10 年：5；30 年：10；
- 单公司财务事实：3–8 credits；
- 多公司比较：5–15 credits；
- 全市场筛选：8–20 credits；
- 事件研究：20–50 credits；
- 深度研究报告：按数据工具 + LLM 预算合并预估。

用户界面应在高成本任务前展示预估；实际失败或系统重试不得重复收费。

## 15.4 数据合作方商业安排

可谈判的模式：

- 固定年度授权费；
- 最低保证 + 超额用户/调用费；
- 订阅收入分成；
- MCP/API 单独收入分成；
- 特定高价值数据包单独计费；
- 数据质量与 SLA 奖惩；
- 双方联合品牌或白标。

推荐避免“所有产品收入统一分成”这种模糊结构，应按数据集、渠道和客户类型建立结算维度，以便未来扩展新数据源。

## 15.5 单位经济模型

单用户月度贡献毛利：

> 订阅与超额收入 − 数据授权分摊 − 数据按量费 − LLM Token − Cloudflare/数据库/搜索 − 支付费 − 直接支持成本

建议目标：

- B2C 付费套餐贡献毛利率长期 > 70%；
- Developer/MCP > 60%，并通过预计算、分页和缓存提升；
- Enterprise 计入实施成本后仍保持正贡献毛利；
- 单次基础研究的边际成本控制在套餐月费的低个位数百分比；
- 高成本用户通过配额、超额收费和模型路由管理，而不是由普通用户补贴。

## 15.6 增长与分发

### 初期获客

- 数据合作方现有客户和品牌渠道；
- 港股研究 SEO：围绕公司、公告、历史指标和事件形成可索引页面；
- 财经内容创作者使用可验证图表和分享卡；
- MCP 目录、AI 开发者社区和连接模板；
- 与券商、媒体和投资教育机构合作；
- “免费研究模板 + 付费深度数据”转化。

### 产品增长循环

1. 用户提出真实研究问题；
2. 工具使用与失败进入匿名化评估集；
3. 改善数据、工具描述、指标和模板；
4. 更好的答案与分享带来新用户；
5. 订阅和 MCP 收入反哺数据覆盖和质量。

# 17. 用户故事与验收场景

## 17.1 Web 用户故事

### US-W01 证券解析

**作为**个人投资者，**我希望**输入“700”“腾讯”或“00700.HK”都能找到同一证券，**以便**不用记住标准格式。

- 候选项显示市场、公司、证券状态和有效日期；
- 存在歧义时不自动猜测；
- URL 和保存记录使用稳定 instrument_id。

### US-W02 近期波动解释

**作为**研究型用户，**我希望**问“为什么最近下跌”，**以便**同时看到行情、公告和基准对照。

- 答案显示数据截止时间和延迟；
- 至少一项价格证据和一项事件/公告证据；
- 明确分开已知事实与市场推断；
- 无证据时不能编造原因。

### US-W03 长期趋势

**作为**长期投资者，**我希望**比较 10–30 年财务与股价，**以便**识别周期。

- 可选择复权口径；
- 财务重述可查看；
- 退市、停牌和缺失样本不被静默删除；
- 图表可保存到研究库。

### US-W04 多股比较

**作为**资深用户，**我希望**比较多只同业股票的增长、估值和回撤，**以便**统一口径判断。

- 币种和单位统一或明确不可比；
- 显示每个指标日期；
- 缺失数据有标识；
- 排序依据可解释。

### US-W05 自然语言筛选

**作为**用户，**我希望**用自然语言写筛选条件，**以便**不用学习复杂语法。

- 执行前可检查并编辑结构化条件；
- 结果显示命中原因；
- 支持保存和重复运行；
- 不允许使用未来数据筛选历史时点。

### US-W06 公告核实

**作为**用户，**我希望**从摘要跳到原文位置，**以便**验证模型说法。

- 引用定位到文档、页/段和摘录；
- 摘录不超过授权范围；
- 文档中的恶意指令不会触发工具调用。

### US-W07 新手解释

**作为**港股新手，**我希望**看到易懂解释，**以便**理解 ROE、自由现金流和复权。

- 提供简明定义和例子；
- 不降低数据准确性；
- 可一键切换专业视图。

### US-W08 保存与重放

**作为**用户，**我希望**保存研究并在未来更新，**以便**知道哪些变化来自新数据。

- 保存证据快照和版本；
- 重跑时显示数据、模型和参数差异；
- 旧报告不会被新数据静默改写。

### US-W09 提醒

**作为**观察列表用户，**我希望**只收到有实质意义的提醒，**以便**减少噪音。

- 支持频率与静默期；
- 相同事件去重；
- 每个提醒带来源；
- 可一键关闭。

### US-W10 高成本任务确认

**作为**付费用户，**我希望**深度报告执行前看到消耗预估，**以便**控制额度。

- 显示预计 credits、数据范围和输出；
- 用户确认后预扣；
- 系统失败自动退还；
- 完成后显示实际消耗。

## 17.2 MCP 用户故事

### US-M01 快速连接

**作为**AI Power User，**我希望**按客户端向导连接 AiphaBee，**以便**十分钟内完成第一次调用。

- 支持复制 endpoint 或一键配置；
- OAuth 页面解释 scope；
- Developer Console 显示连接状态；
- 首次调用有可复制示例。

### US-M02 结构化历史行情

**作为**外部 Agent，**我希望**得到有 Schema 的历史数据，**以便**可靠计算而不是解析自然语言。

- 返回 `structuredContent`；
- 日期、币种、复权和时区明确；
- 大结果使用游标；
- 每页响应都遵守套餐总量。

### US-M03 数据血缘

**作为**开发者，**我希望**查询一个数值的来源，**以便**在自己的输出中引用。

- `get_data_lineage` 返回数据集、源记录、版本和方法；
- 不泄露合作方不允许暴露的内部信息；
- 来源 ID 在保留期内稳定。

### US-M04 权限不足

**作为**客户端，**我希望**权限不足时收到机器可处理的错误，**以便**提示用户授权或缩小请求。

- 返回 `DATA_NOT_LICENSED` 或 `SCOPE_DENIED`；
- 错误中包含可允许的替代范围；
- 不返回部分敏感字段。

### US-M05 配额与对账

**作为**Developer 用户，**我希望**看到每次调用消耗，**以便**控制成本。

- 响应有 usage 摘要；
- Console 可按日期、工具和 Key 查看；
- 账单与调用 request_id 对应；
- 系统重试不重复计费。

### US-M06 版本稳定

**作为**集成开发者，**我希望**工具升级不突然破坏应用。

- 工具版本可查询；
- 破坏性变化发布新 major；
- 弃用有公告与迁移示例；
- 旧版本在约定窗口内可用。

### US-M07 防止无限抓取

**作为**平台运营方，**我希望**限制批量抓取，**以便**保护数据授权和成本。

- 游标、行数、时间、证券和并发多层限制；
- 异常模式触发风控；
- 多 Key/多账户关联分析；
- 企业合法批量需求通过专门套餐解决。

### US-M08 外部客户端可解释

**作为**终端用户，**我希望**外部 AI 能告诉我数据延迟与来源，**以便**不误把旧数据当实时。

- Tool description 明确要求客户端展示 `as_of` 和 delay；
- 返回结构中该字段不可省略；
- 无法确定时返回警告而非空值。

## 17.3 运营、数据与合规用户故事

### US-O01 字段授权

运营可配置“某数据集仅 Web Pro 可展示，MCP Developer 只能返回聚合值”，且变更有审批、版本和生效时间。

### US-O02 数据隔离

数据质量人员可暂停一个字段或证券的对外服务，不影响其他正常数据；用户收到 `DATA_QUALITY_HOLD`。

### US-O03 Run 调查

支持人员可用 request_id 查看工具、数据版本、错误和计费，但默认看不到不必要的用户敏感内容。

### US-O04 模型/工具 Kill Switch

合规或工程负责人可立即停用某模型、工具、数据集或功能，并自动切换安全降级体验。

### US-O05 更正通知

数据人员修正重大错误后，系统识别受影响报告并生成通知与重跑入口。

### US-O06 合作方结算

财务/运营可按数据集、渠道、套餐、用户和调用量导出对账，而不依赖人工日志拼接。

# 20. 风险与缓解

| 风险 | 影响 | 可能性 | 主要缓解 |
|---|---|---:|---|
| 数据合作方无完整再分发权 | 产品无法合法提供 MCP/部分 Web 数据 | 高 | Gate 0 权利矩阵；默认拒绝；分层数据包 |
| 功能构成投资建议/Type 4 | 监管、下架和声誉风险 | 中高 | 限制个性化建议；书面法律意见；持牌合作路线 |
| 30 年数据口径不一致 | 长期分析误导 | 高 | 证券主表、版本、复权、重述和黄金样本 |
| 模型生成虚假数字 | 用户信任和合规事故 | 中高 | 工具证据绑定；生成后校验；禁止模型凭记忆报数 |
| MCP 被用于批量抓取 | 数据授权和成本失控 | 高 | 多维限流、credits、异常检测、企业批量套餐 |
| 合作方或模型供应商故障 | 服务中断 | 中 | 数据快照、SLA、fallback、Workflows 重试、状态提示 |
| Cloudflare 单请求边界 | 深度任务失败或断线 | 中 | 交互/长任务分界；Workflows/DO；可恢复 task_id |
| 产品过于复杂 | 新手流失 | 中 | 模板、分层信息、默认任务而非空聊天框 |
| 套餐成本错配 | 毛利为负 | 中高 | 加权 credits、预算、预计算、模型路由、价格实验 |
| 广告/券商合作影响中立 | 品牌受损 | 中 | 商业与研究隔离、披露、不按佣金排序 |
| MCP 客户端兼容差异 | 连接失败和支持成本 | 中 | 官方协议、兼容测试、文本回退、版本策略 |
| 用户隐私泄露 | 法律与信任风险 | 中 | 最小化、加密、供应商评估、可删除记忆和审计 |

# 22. 决策记录

| 决策 | 状态 | 理由 |
|---|---|---|
| Agent Runtime = AI SDK v7 + Hono on Workers | 已决策 | 同栈、边缘、无沙箱需求、适合高并发 |

| MCP 原子工具默认不嵌套 LLM | 本 PRD 建议 | 降低成本、延迟和不可预测性 |
| Postgres 为主真值源 | 本 PRD 建议 | 权益、计费、审计和金融时序关系复杂 |
| Web 与 MCP 权利分离 | 必须 | 机器可读再分发风险和合同义务不同 |
| MVP 不做交易和个性化建议 | 本 PRD 建议 | 降低牌照和用户安全风险 |
| 延迟/EOD 优先 | 本 PRD 建议 | 降低实时授权成本和上线复杂度 |
